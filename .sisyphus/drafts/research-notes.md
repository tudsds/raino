## 1. Supabase pgvector

### Official Source

- Docs: https://supabase.com/docs/guides/ai/vector-columns
- RAG with RLS: https://supabase.com/docs/guides/ai/rag-with-permissions
- Semantic search: https://supabase.com/docs/guides/ai/semantic-search

### What It Confirms

- Enable via `create extension if not exists vector with schema extensions`
- Index types: HNSW (recommended, <1M rows) vs IVFFlat (>1M rows)
- Distance operators: `<#>` inner product, `<=>` cosine, `<->` euclidean
- RLS works with vector search — operators respect policies automatically
- Vector ops require `supabase.rpc()` — PostgREST doesn't understand pgvector operators

### Why It Matters to Raino

- Raino already has HNSW index + match_documents RPC ✅
- RLS on vector tables means per-user document isolation is possible
- Must use RPC pattern (already implemented) for vector queries
- Dimension must match embedding model: current 384 → need 1536 for OpenAI

### Recommended Setup

```sql
-- Dimensions match embedding model
-- text-embedding-3-small = 1536, gte-small = 384
CREATE INDEX ON embeddings USING hnsw (vector vector_cosine_ops) WITH (m=32, ef_construction=80);
```

### RLS Optimization Gotcha

- Avoid JOINs in RLS policies (causes 9000ms queries)
- Use IN subqueries instead (~20ms, 99.78% improvement)

## 2. Supabase Auth (Magic Link + SSR)

### Official Source

- Magic link: https://supabase.com/docs/guides/auth/passwordless-auth
- SSR: https://supabase.com/docs/guides/auth/server-side/rendering

### What It Confirms

- Magic link flow: email → POST /auth/v1/magiclink → email with link → callback exchange
- Three client types: browser (createBrowserClient), server component (createServerClient with cookies), middleware (createServerClient with getAll/setAll)
- Server components: `cookies()` is async, can only READ
- Middleware: Can READ and SET cookies
- Session refresh: MUST use `getUser()` not `getSession()` in middleware

### Why It Matters to Raino

- Raino already implements three-client pattern correctly ✅
- Middleware refreshes session on every request ✅
- Auth callback exchanges code for session ✅
- User provisioning on first sign-in ✅

### Gotcha

- `getUser()` sends the JWT to Supabase for validation (network call)
- `getSession()` only checks local cookies (no validation)
- Always use `getUser()` in server contexts for security

## 3. Kimi K2.5 / Moonshot API

### Official Source

- Main docs: https://platform.moonshot.ai/docs/
- API reference: https://platform.moonshot.ai/docs/api/chat
- Pricing: https://platform.moonshot.ai/docs/pricing/chat-k25
- Rate limits: https://platform.moonshot.ai/docs/pricing/limits

### What It Confirms

- Base URL: `https://api.moonshot.ai/v1` (global) or `https://api.moonshot.cn/v1` (China)
- **Fully OpenAI SDK compatible** — use `openai` npm package with custom baseURL
- Models: `kimi-k2.5` (flagship, 256K context), `kimi-k2-turbo-preview` (fast), `kimi-k2-thinking` (reasoning)
- JSON mode: `response_format: { type: "json_object" }` (must prompt for JSON explicitly)
- Function calling: Standard tools interface
- Pricing: $0.60/M input (cache miss), $0.10/M input (cache hit), $3.00/M output

### Why It Matters to Raino

- Raino uses `openai` SDK pointed at Moonshot — correct pattern ✅
- `packages/llm/src/` wraps this correctly with retry logic ✅
- For development/testing: use `kimi-k2-turbo-preview` (60-100 tok/s, cheaper)

### CRITICAL: No Embedding API

- Moonshot does NOT expose `/v1/embeddings`
- Raino CANNOT use Moonshot for embeddings
- Must use separate provider: OpenAI `text-embedding-3-small` recommended ($0.02/1M tokens)

### Gotchas

- Temperature: must be 1.0 with thinking=True, 0.6 with thinking=False for kimi-k2.5
- Rate limits use `max_completion_tokens` budget, not actual tokens
- Keys not interchangeable between api.moonshot.ai and api.moonshot.cn
- `max_tokens` deprecated — use `max_completion_tokens`

## 4. DigiKey API

### Auth

- OAuth2 client_credentials grant (server-to-server)
- `POST /v1/oauth2/token` with `grant_type=client_credentials`
- Token cached with expiry, 401 triggers refresh

### Key Endpoints

- `GET /products/v4/search/keyword` — keyword search
- `GET /products/v4/product/details` — part by DigiKey number
- `GET /products/v4/categories` — category tree

### Why It Matters to Raino

- Raino uses client_credentials (not user OAuth) — correct for server-side ✅
- `DIGIKEY_REDIRECT_URI` in env is misleading — not needed for client_credentials
- Rate limits: 429 should trigger backoff (currently throws immediately)

## 6. JLCPCB API

### Auth

- MD5 signature: `MD5(timestamp + nonce + secretKey)`
- Params: `key=accessKey`, `timestamp`, `nonce`, `signature`
- Base URL: `https://ips.lcsc.com/rest/wmsc2agent/`

### Key Endpoints

- `GET /product/search` — keyword search
- `GET /product/detail` — part by product code

### Why It Matters to Raino

- Raino's `RealJLCPCBAdapter` uses correct auth shape ✅
- MD5 is JLCPCB's required protocol — not a Raino security choice
- Document as known characteristic (mitigated by HTTPS transport)

## 8. OpenAI Embeddings (Recommended Provider)

### Recommended Model

- `text-embedding-3-small`: 1536 dimensions, $0.02/1M tokens
- Best cost/quality tradeoff for RAG

### Integration Pattern

```typescript
import OpenAI from 'openai';
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
const response = await openai.embeddings.create({
  model: 'text-embedding-3-small',
  input: text,
});
const embedding = response.data[0].embedding; // number[] of length 1536
```

### Why It Matters to Raino

- Moonshot has NO embedding API — must use OpenAI or other provider
- pgvector dimension must change from 384 → 1536
- Cost is negligible: $0.02/1M tokens for ingestion + queries
